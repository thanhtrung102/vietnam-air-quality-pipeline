## ADR-001: Cloud Platform — AWS over GCP

### Context

The pipeline must ingest data from the OpenAQ public S3 archive (hosted in AWS S3, us-east-1) and store, transform, and visualise it. Two viable cloud platforms were evaluated: AWS (ap-southeast-1) and GCP (asia-southeast1).

### Decision

AWS was selected as the sole cloud platform.

### Rationale

The highest-cost operation in this pipeline is the bulk sync of the OpenAQ public archive. That archive resides in AWS S3 (us-east-1). An intra-AWS S3-to-S3 sync using `aws s3 sync` benefits from AWS internal transfer pricing: cross-region egress from us-east-1 to ap-southeast-1 is charged at $0.02/GB (requester-pays), which is substantially cheaper than exfiltrating the same data to GCP. AWS also offers the Athena + Glue serverless warehouse pattern with a mature, community-supported dbt adapter (`dbt-athena-community`). QuickSight integrates with Athena natively without a connector or intermediate storage layer. The IAM user `terraform-admin` and associated credentials already exist in the target AWS account, reducing bootstrap cost.

GCP's BigQuery would have offered a stronger native SQL experience, better INFORMATION_SCHEMA introspection, and tighter streaming integration via Pub/Sub. However, routing all S3 archive traffic through GCP would incur cross-cloud egress at $0.08–$0.12/GB — a 4–6× cost penalty on every backfill and incremental sync.

### Consequences

- All infrastructure is AWS-native; no multi-cloud complexity.
- Cross-region S3 egress cost (us-east-1 → ap-southeast-1) is accepted as necessary.
- Athena query syntax and Glue catalogue conventions must be followed throughout.
- GCP's more ergonomic BI tooling (Looker Studio free tier) is unavailable; QuickSight per-user licensing applies.

## ADR-002: Batch Ingestion — S3 Sync over API

### Context

Historical air quality data (2022–present) for Vietnamese stations must be ingested into the project S3 bucket. Two approaches were evaluated: (a) syncing directly from the OpenAQ public S3 archive using `aws s3 sync`, and (b) fetching measurements record-by-record from the OpenAQ v3 REST API with pagination.

### Decision

`aws s3 sync` from the public archive was selected for all historical batch loads.

### Rationale

The OpenAQ public archive stores data as gzip-compressed Parquet files, already partitioned by `locationid/year/month`. These files are the highest-quality representation of the data — they are generated by OpenAQ's own ETL, validated, and contain the complete column set. Fetching via the REST API would require: iterating through paginated JSON responses (up to millions of records per location per year), deserialising JSON, re-serialising to Parquet, and managing pagination cursors. This is computationally expensive, API-rate-limited, and produces lower-quality output (the API does not expose all archive columns such as `is_mobile` and `source_name`).

`aws s3 sync` transfers at S3 internal throughput (100–500 MB/s effective), making a three-year backfill of all Vietnamese stations (estimated 5–20 GB) completable in minutes. It is also inherently idempotent: re-running the sync skips files whose ETag and size match, making retry and incremental daily updates safe and cheap.

### Consequences

- Requester-pays egress fees apply to all S3 sync operations.
- The 72-hour archive lag is accepted for the batch path; the streaming bridge (ADR-003) fills this window.
- Vietnamese station IDs must be known before running the sync; a station discovery step is required during initial setup.
- Future backfill extension to other countries requires only adding location IDs to the include filter.

## ADR-003: Two-Source Architecture — Archive for Batch, API for Streaming

### Context

No single OpenAQ data source satisfies all requirements simultaneously. The public S3 archive is complete and efficient but lags real time by ~72 hours. The v3 REST API is near-real-time but rate-limited, JSON-only, and unsuitable for bulk history.

### Decision

A two-source architecture was adopted: `s3 sync` from the archive for all data older than 72 hours, and periodic API polling for the recency window.

### Rationale

The 72-hour handover boundary is a natural seam. Data older than 72 hours is guaranteed to be in the archive and should be fetched from there (cheap, complete, Parquet). Data younger than 72 hours is not yet in the archive and must come from the API. The two prefixes (`raw/batch/` and `raw/stream/`) are both catalogued in the same Glue database, and a Athena `UNION ALL` view in the staging models merges them transparently. dbt's `stg_measurements_batch` and `stg_measurements_stream` models apply identical column normalisation before the union, ensuring the mart sees a homogeneous fact table regardless of data origin.

A single-source design using only the API would require storing 72 hours of overlap in the stream prefix and would lose the efficiency of the Parquet archive. A single-source design using only the archive would make the dashboard data perpetually 72 hours stale, defeating the goal of timely AQI awareness.

### Consequences

- Two Glue Crawlers and two Athena external tables must be maintained.
- The staging layer must handle the union and deduplicate any overlap in the 72-hour window.
- A `data_source` column (`batch` or `stream`) is preserved in the mart for provenance tracking.
- If OpenAQ reduces their archive lag in the future, the streaming window can be shortened without structural changes.

## ADR-004: Athena over Redshift Serverless

### Context

A SQL warehouse is required to serve dbt transformations and QuickSight queries over data stored in S3. Two serverless options were evaluated: Amazon Athena and Amazon Redshift Serverless.

### Decision

Amazon Athena was selected.

### Rationale

Athena is optimised for the exact workload this pipeline produces: S3-native data, infrequent analytical queries, schema-on-read, and no concurrent interactive users. It charges $5 per TB scanned with zero idle cost. For the expected query volume — daily dbt runs (< 1 TB/day), one or two ad-hoc explorations, and two QuickSight SPICE refreshes — the monthly Athena cost is estimated at $5–$20/month. Redshift Serverless would cost a minimum of $0.36/RPU-hour; even at the minimum capacity of 8 RPUs, keeping the endpoint available for a 4-hour daily window costs ~$43/month. At the expected scale this is roughly 2–8× more expensive for equivalent query performance.

Athena also requires no VPC, no subnet configuration, and no cluster state management. The Glue Data Catalogue serves as the shared metadata store, eliminating the need to maintain a separate Redshift schema definition.

Redshift Serverless would be the better choice if the workload included concurrent BI users querying the live table (not SPICE-cached), complex window functions over very large datasets, or sub-second query latency requirements. None of these apply here.

### Consequences

- Query performance is bounded by S3 I/O; complex queries over years of data may take 30–90 seconds.
- Partition projection must be correctly configured to avoid full-table scans.
- Athena does not enforce schemas; data quality must be enforced upstream (in ingestion) or in dbt tests.
- Result reuse is limited to 7 days; QuickSight SPICE caching is required to avoid repeated Athena scans for dashboard loads.

## ADR-005: Kinesis over Kafka (MSK)

### Context

An optional real-time streaming path is required for sub-minute latency alerting use-cases. Two managed streaming options were evaluated: Amazon Kinesis Data Streams and Amazon MSK (Managed Streaming for Apache Kafka).

### Decision

Amazon Kinesis Data Streams was selected for the streaming path.

### Rationale

Kinesis is fully managed with no broker, topic, or ZooKeeper configuration. A single shard handles 1 MB/s write and 2 MB/s read throughput — more than sufficient for the expected volume (a few hundred records per minute from ~20 Vietnamese monitoring stations). Kinesis integrates directly with Kinesis Data Firehose for S3 delivery and with Lambda for event-driven processing, covering both the persistence and alerting use-cases without additional infrastructure.

Amazon MSK would provide higher throughput ceiling, consumer group semantics, Kafka Streams processing, and richer ecosystem tooling (Kafka Connect, ksqlDB). However, MSK requires at least two broker instances in two availability zones ($0.21/hour per broker minimum = ~$300/month), which is disproportionate for this throughput tier. MSK also introduces Kafka-specific operational knowledge (topic management, offset management, consumer group lag monitoring) that adds complexity without delivering proportionate value at this scale.

If the pipeline is extended to aggregate OpenAQ data globally (hundreds of countries, thousands of stations), the higher throughput and consumer group semantics of MSK would justify the operational cost.

### Consequences

- Kinesis stream `openaq_stream` is provisioned at 1 shard; this can be scaled to N shards if throughput exceeds 1 MB/s.
- Kinesis records are retained for 24 hours by default (configurable to 7 days); downstream consumers must process within the retention window.
- Kinesis does not support topic-level consumer group offsets; each Firehose/Lambda consumer maintains its own shard iterator position.
- The streaming path is provisioned but not activated in the initial deployment; it is reserved for Phase 2 (operational alerting).

## ADR-006: Kestra over Airflow (MWAA)

### Context

An orchestration layer is required to schedule and coordinate: S3 archive sync, Glue Crawler triggering, API polling, dbt runs, and data quality checks. Two options were evaluated: Kestra (self-hosted) and Amazon MWAA (Managed Workflows for Apache Airflow).

### Decision

Kestra was selected as the orchestration platform.

### Rationale

Kestra defines workflows in YAML, which is version-controlled naturally in the repository alongside the code it orchestrates. Every required integration — `aws s3 sync` (via shell task), Glue Crawler trigger (via AWS plugin), HTTP request (for OpenAQ API), Python script execution (for Parquet writing), and dbt CLI invocation — is covered by Kestra's built-in task plugins. Kestra's UI provides flow visualisation, execution history, and log inspection without additional configuration.

MWAA (managed Airflow) would require a minimum environment configuration of ~$0.49/hour ($350/month) for the scheduler and web server, before any worker costs. Airflow's Python DAG model is more expressive for complex branching logic, supports XCom for inter-task data passing, and has a larger plugin ecosystem. However, for this pipeline's linear workflow (sync → crawl → poll → dbt run → test), Airflow's expressivity is not needed, and its operational cost is prohibitive compared to Kestra running on a small ECS task or EC2 instance.

### Consequences

- Kestra is self-managed; infrastructure (ECS task or EC2) must be provisioned and patched.
- Kestra YAML flows are the authoritative orchestration definition and must be kept in sync with code changes.
- If the pipeline grows to dozens of complex interdependent DAGs, migration to Airflow may be warranted.
- Kestra's secrets management (via environment injection or its built-in vault) must be used instead of hardcoding credentials in flow YAML.

## ADR-007: Partition and Cluster Key Choices — `measurement_date` / `(parameter, location_id)`

### Context

The mart table `mart_daily_air_quality` in Athena (Iceberg) must support efficient queries for: (a) time-range trends (last 30 / 90 / 365 days, full three-year history), (b) pollutant-specific analysis (`WHERE parameter = 'pm25'`), and (c) station-specific drill-down (`WHERE location_id = ?`). Partition and sort order choices directly determine query cost and latency.

### Decision

- **Partition key:** `measurement_date` (DATE)
- **Iceberg sort order (cluster keys):** `parameter` ASC, `location_id` ASC

### Rationale

**Partition on `measurement_date`:**
Every analytical query in the dashboard filters on a date range. Partitioning on `measurement_date` means Athena's partition pruner eliminates all partitions outside the requested window before scanning a single byte of data. A "last 30 days" query reads exactly 30 partitions. Alternative partition candidates were considered:

- Partitioning on `location_id` would benefit station-specific queries but produce hundreds of small partitions, increasing Glue catalogue overhead and degrading date-range query performance (Athena must open all location partitions to satisfy a date filter).
- Partitioning on `parameter` would require scanning 6 partitions for dashboard views that show all pollutants.
- A composite partition on `(measurement_date, parameter)` would reduce scan for single-pollutant queries but more than double the partition count and complicate the SPICE refresh schedule.

Daily date partitioning yields ~1,095 partitions for three years — well within Athena's efficient range.

**Sort order on `(parameter, location_id)`:**
Within each date partition, data is physically ordered first by pollutant code then by station. Dashboard queries typically combine a date range filter with a pollutant filter (`WHERE parameter = 'pm25' AND measurement_date BETWEEN ...`). With this sort order, the Parquet row-group statistics allow the Iceberg reader to skip all row groups where `parameter` is not `pm25`, dramatically reducing the bytes read per partition. `location_id` as the secondary sort key benefits city-comparison queries that combine a date range, a pollutant, and a set of station IDs.

### CAP Theorem Justification — Athena as a CP System

Athena is a **Consistency + Partition Tolerance (CP)** system in the context of this pipeline. When Athena executes a query, it reads from S3, which provides strong read-after-write consistency (as of December 2020, all S3 operations are strongly consistent). A query submitted after a dbt `run` completes is guaranteed to see the fully written Iceberg snapshot — Athena will never return a partial or stale view of a committed write. This is the Consistency guarantee.

Athena tolerates network partitions between its query engine nodes and S3 because S3 itself is designed for Partition Tolerance — objects are replicated across multiple AZs and remain available through AZ-level failures. This is the Partition Tolerance guarantee.

Athena sacrifices **Availability** in the CAP sense: if the Athena service itself or its metadata plane (Glue) experiences an outage, queries fail rather than returning potentially stale results. This is the correct trade-off for an analytics warehouse where incorrect data would be worse than no data (a stale dashboard showing PM2.5 as "Good" during an actual pollution event is more harmful than a temporarily unavailable dashboard).

The Iceberg table format reinforces the CP choice: each dbt `run` produces an atomic Iceberg snapshot commit. Queries either see the previous complete snapshot or the new complete snapshot — never a half-written state. This snapshot isolation is the Iceberg analogue of ACID transaction atomicity.

### Consequences

- Daily date partitions accumulate at the rate of one per day; a three-year history produces 1,095 partitions — manageable for Glue and Athena.
- The Iceberg sort order is enforced at write time by dbt's `+sort` config; queries that do not filter on `parameter` will not benefit from row-group skipping.
- Time-travel queries (`SELECT ... FOR SYSTEM_TIME AS OF ...`) are available through Iceberg but require careful snapshot retention management to avoid unbounded S3 growth.
- If real-time queries (latency < 1 minute) become a requirement, the CP guarantee of Athena means accepting that availability drops during service disruptions; a separate HA alerting path (Lambda + SNS) would be needed for SLA-critical alerting.

## ADR-008: Orchestration — EventBridge Scheduler + Lambda over Kestra

### Context
Kestra requires Docker Compose to run. Docker Desktop is unavailable on the
Windows build machine. Two alternatives were evaluated: (a) AWS EventBridge
Scheduler + Lambda, and (b) deferring orchestration and triggering scripts
manually.

### Decision
AWS EventBridge Scheduler + Lambda was selected.

### Rationale
EventBridge Scheduler provides cron-based invocation semantics identical to
Kestra's scheduled flows, with zero local infrastructure dependency. Lambda
cold start under 2 seconds is acceptable for a daily batch job and a
30-minute streaming poll. Both are fully managed, monitored via CloudWatch
Logs, and provisioned by the existing Terraform configuration. Manual
triggering was rejected because it would break the reproducibility criterion
— the pipeline would not be self-sustaining without human intervention.

### Consequences
- No Kestra UI for flow visualisation; execution history is in CloudWatch Logs.
- Lambda has a 15-minute maximum timeout; the batch sync must complete within
  this window. At current data volumes (21 stations, monthly delta) this is
  comfortable. If the station list grows significantly, the batch job will
  need to be restructured as parallel Lambda invocations.
- The Kestra docker-compose.yml is retained in the repository as reference
  documentation for the original design intent.
