```sql
-- AQI correctness: Hanoi 3-year mean PM2.5 should be 35–50 µg/m³
SELECT city,
       ROUND(AVG(avg_value), 1) AS mean_pm25_corrected,
       ROUND(AVG(composite_aqi), 0) AS mean_composite_aqi
FROM openaq_mart.mart_daily_aqi a
JOIN openaq_mart.mart_daily_air_quality q
  ON a.location_id = q.location_id AND a.measurement_date = q.measurement_date
WHERE q.parameter = 'pm25' AND q.is_outlier_station = 0
GROUP BY city;
-- Expected: Hanoi ~40 µg/m³, HCMC ~21 µg/m³
```

```sql
-- Weekend classification correctness (Saturday=6, Sunday=7 in ISO)
SELECT day_type, hour_of_day, ROUND(AVG(avg_value), 1) AS avg_pm25
FROM openaq_mart.mart_diurnal_profile
WHERE parameter = 'pm25' AND city = 'Hanoi'
GROUP BY day_type, hour_of_day
ORDER BY day_type, hour_of_day;
-- Weekend morning peak should be ~1 hour later than weekday (traffic offset)
```

## 5.5.2 Forecast Lambda Container {#forecast-lambda}

The `forecast_generate` Lambda runs as a Docker container image stored in ECR. It was skipped during the first `terraform apply` because the ECR repository existed but had no image.

*[VI: Lambda `forecast_generate` chạy như là Docker container image được lưu trữ trong ECR.]*

### Step 1: Build the Docker image

```bash
cd lambda/forecast_generate/
docker build -t openaq-forecast-generate .
```

Expected build time: ~3–5 minutes (downloads python:3.12-slim, installs statsmodels, pandas, pyarrow, boto3).

### Step 2: Push to ECR

```bash
AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
AWS_REGION=ap-southeast-1

# Authenticate Docker to ECR
aws ecr get-login-password --region $AWS_REGION \
  | docker login --username AWS \
    --password-stdin $AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com

# Tag and push
docker tag openaq-forecast-generate:latest \
  $AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/openaq-forecast-generate:latest

docker push \
  $AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/openaq-forecast-generate:latest
```

### Step 3: Wire the image to Lambda via Terraform

```bash
IMAGE_URI=$AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/openaq-forecast-generate:latest

cd terraform/
terraform apply -var="forecast_lambda_image_uri=$IMAGE_URI"
```

This second apply modifies one resource (`aws_lambda_function.forecast_generate`) and wires the ECR image. Type `yes`.

### Step 4: Register the external forecast table (one-time)

The forecast Lambda writes Parquet to S3 and expects an Athena external table registered once per environment. Run this DDL in the Athena console (workgroup: `openaq_workgroup`), replacing `{bucket}` with your actual bucket name:

```sql
-- Full DDL is in: transform/setup/create_forecast_table.sql
-- Replace {bucket} before running
CREATE EXTERNAL TABLE IF NOT EXISTS openaq_mart.mart_daily_forecast (
  location_id     bigint,
  location_name   string,
  city            string,
  forecast_date   date,
  model           string,
  pm25_forecast   double,
  ci_lower        double,
  ci_upper        double,
  generated_at    timestamp
)
STORED AS PARQUET
LOCATION 's3://{bucket}/processed/openaq_mart/mart_daily_forecast/'
TBLPROPERTIES ('parquet.compression'='SNAPPY');
```

### Step 5: Invoke the forecast Lambda

```bash
aws lambda invoke \
  --function-name openaq_forecast_generate \
  --payload '{}' \
  --cli-binary-format raw-in-base64-out \
  /tmp/forecast_response.json

cat /tmp/forecast_response.json
```

Expected response:
```json
{
  "statusCode": 200,
  "stations_forecasted": 3,
  "records_written": 21,
  "rmse_hanoi": 11.8,
  "rmse_hcmc": 6.4
}
```

> **`stations_forecasted: 3`** — only 3 of 21 stations have readings within the last 90 days. The remaining 18 stopped reporting to OpenAQ in 2024–2025. The Lambda skips stale stations automatically.

### Verify forecast in Athena

```sql
SELECT location_name, forecast_date, ROUND(pm25_forecast, 1) AS pm25_fcst,
       ROUND(ci_lower, 1) AS ci_low, ROUND(ci_upper, 1) AS ci_high
FROM openaq_mart.mart_daily_forecast
ORDER BY location_name, forecast_date;
-- Expected: 21 rows (3 stations × 7 days), forecast dates starting tomorrow
```

## 5.5.3 Security & IAM Design {#iam} | Bảo mật và Thiết kế IAM

The project applies four security principles throughout:

*[VI: Dự án áp dụng bốn nguyên tắc bảo mật: IAM ít đặc quyền, bí mật trong Secrets Manager, bảo vệ XSS, và giới hạn quét Athena.]*

### Principle 1: Least-privilege IAM per Lambda

A single execution role (`openaq_lambda_role`) is shared, but the **inline policy is resource-scoped** — not wildcards. Each permission covers only the specific ARNs the Lambda actually needs:

| Permission block | Actions | Resource scope |
|-----------------|---------|---------------|
| S3 project bucket | `PutObject`, `GetObject`, `DeleteObject` | `raw/*`, `processed/*`, `athena-results/*` prefixes only |
| S3 OpenAQ archive | `GetObject` | `arn:aws:s3:::openaq-data-archive/records/csv.gz/*` only |
| Glue catalog | `GetDatabase`, `GetTable`, `GetPartitions` | `openaq_raw` and `openaq_mart` databases only |
| Athena | `StartQueryExecution`, `GetQueryExecution`, `GetQueryResults` | `openaq_workgroup` only |
| Kinesis | `PutRecord`, `PutRecords` | `openaq_stream` ARN only |
| Secrets Manager | `GetSecretValue` | `openaq/api_key` ARN only |
| CloudWatch | `PutMetricData` | Namespace `OpenAQ/Pipeline` only |
| SNS | `Publish` | `openaq_alerts` ARN only |

This means a compromised `aqi_api` Lambda cannot write to Kinesis, and a compromised `batch_sync` Lambda cannot call Secrets Manager.

### Principle 2: Secrets Manager — no hardcoded keys

`OPENAQ_API_KEY` is **not** in Lambda environment variables. The streaming producer resolves it at runtime:

```python
# lambda/streaming/kinesis_producer.py
def _get_api_key() -> str:
    """Fetch API key from Secrets Manager; fall back to env var for local testing."""
    client = boto3.client("secretsmanager", region_name=os.environ["AWS_REGION"])
    secret = client.get_secret_value(SecretId="openaq/api_key")
    return secret["SecretString"]
```

The key is cached in the module-level variable — one Secrets Manager call per Lambda warm container, not per invocation.

### Principle 3: Athena scan guard

The `openaq_workgroup` enforces a **10 GB per-query scan limit**. Any query that would exceed this is cancelled before it runs. At current data volume (800 MB) this prevents accidental cost spikes from un-partitioned queries while leaving safe headroom.

### Principle 4: XSS protection in the dashboard

Station data from the API (names, categories, values) is injected into the Leaflet popup via `innerHTML`. Without escaping, a malicious station name containing `<script>` tags would execute in the user's browser.

The `escHtml()` function in `dashboard/index.html` sanitises all seven user-visible fields before insertion:

```javascript
function escHtml(s) {
    if (s == null) return "";
    return String(s)
      .replace(/&/g, "&amp;")
      .replace(/</g, "&lt;")
      .replace(/>/g, "&gt;")
      .replace(/"/g, "&quot;")
      .replace(/'/g, "&#39;");
}

// Applied to every field in buildPopup():
const name = escHtml(props.location_name);
const aqi  = escHtml(props.composite_aqi);
// ...
```

### CloudWatch Operational Monitoring

Two alarms are active after deployment:

| Alarm | Metric | Threshold | Action |
|-------|--------|-----------|--------|
| `OpenAQ-ForecastRMSE-High` | `OpenAQ/Pipeline ForecastRMSE` | > 25 µg/m³ for 3 consecutive days | SNS email to `alert_email` |
| `OpenAQ-MissingStations` | `OpenAQ/Pipeline ActiveStations` | < 1 station with data in 24 h | SNS email to `alert_email` |

To test an alarm manually:

```bash
aws cloudwatch put-metric-data \
  --namespace "OpenAQ/Pipeline" \
  --metric-name ForecastRMSE \
  --value 30.0 \
  --region ap-southeast-1
```

After 3 such publications you should receive an SNS email alert.
