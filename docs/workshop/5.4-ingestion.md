This section loads all historical and near-real-time data into S3 and validates it in Athena.

*[VI: Phần này tải tất cả dữ liệu lịch sử và gần thời gian thực vào S3 và xác thực trong Athena.]*

## 5.4.1 Historical Batch Sync | Đồng bộ hóa lịch sử

The `sync_historical.sh` script copies all OpenAQ CSV.GZ files for the 21 Vietnamese stations from the OpenAQ public S3 archive to your project bucket. It uses `aws s3 sync`, which is **idempotent** — re-running when files already exist produces zero copies.

*[VI: Script `sync_historical.sh` sao chép tất cả file CSV.GZ OpenAQ cho 21 trạm Việt Nam từ kho lưu trữ S3 công khai của OpenAQ vào bucket của bạn.]*

```bash
# Export your bucket name first
export S3_BUCKET_NAME="openaq-pipeline-yourname"

bash ingestion/historical/sync_historical.sh
```

Expected terminal output (per station-year):
```
syncing  locationid=2178988  year=2023 ...
syncing  locationid=2178988  year=2024 ...
...
==================================================
 Done — success=84  failed=0
==================================================
```

> **Success count explained:** The script iterates over 21 stations × 4 years (2023–2026), so `success=84` when all complete without error.

> **Duration:** The full sync takes 10–20 minutes depending on your network. All files are in `us-east-1` (OpenAQ archive); your bucket is in `ap-southeast-1` — cross-region egress applies at ~$0.02/GB. Total archive size is ~800 MB uncompressed across all 21 stations and 3 years.

> **Dry-run option:** `bash ingestion/historical/sync_historical.sh --dry-run` prints what would be synced without transferring any data.

### Verify in Athena

```sql
-- Total row count (~900,000 expected)
SELECT COUNT(*) AS total_rows
FROM openaq_raw.batch;

-- Date range
SELECT MIN(year) AS min_year, MAX(year) AS max_year,
       COUNT(DISTINCT locationid) AS station_count
FROM openaq_raw.batch;
-- Expected: min_year=2023, max_year=2026, station_count=21

-- Partition pruning proof — scan << 800 MB
SELECT parameter, COUNT(*) AS cnt, ROUND(AVG(CAST(value AS DOUBLE)), 2) AS avg_value
FROM openaq_raw.batch
WHERE year = 2025 AND month = 1
GROUP BY parameter
ORDER BY cnt DESC;
```

Check **Data scanned** in the Athena query stats panel. The date-filtered query should show **< 5 MB** — not the full 800 MB table.

## 5.4.2 Streaming Producer | Nhà sản xuất luồng dữ liệu

The streaming producer Lambda was provisioned and scheduled by Terraform in [Step 5.3.1](5.3-storage-catalog.md). It runs automatically every 30 minutes. To trigger it manually for testing:

*[VI: Lambda streaming producer đã được cung cấp và lên lịch bởi Terraform. Nó chạy tự động mỗi 30 phút.]*

```bash
aws lambda invoke \
  --function-name openaq_streaming_producer \
  --payload '{}' \
  --cli-binary-format raw-in-base64-out \
  /tmp/stream_response.json

cat /tmp/stream_response.json
```

Expected response:
```json
{"success": 42, "failed": 0}
```

> **`records_put` may be 0** if OpenAQ has no new readings in the last 30 minutes — this is normal for stations that report hourly. Try invoking at the top of the hour.

### Verify CloudWatch Logs

```bash
aws logs tail /aws/lambda/openaq_streaming_producer --since 5m
```

Look for:
```
[INFO] sensor cache built — 42 sensors across 21 stations
[INFO] fetched 42 valid reading(s) (sentinel -999.0 excluded)
[INFO] published — success=42  failed=0  stream=openaq_stream
```

### Verify Firehose delivery

Wait 5 minutes (Firehose buffer interval), then check:

```bash
aws s3 ls s3://openaq-pipeline-yourname/raw/stream/ --recursive | tail -5
```

You should see GZIP files under `raw/stream/{yyyy}/{MM}/{dd}/{HH}/`.

## 5.4.3 Weather Backfill | Nạp dữ liệu thời tiết

This step backfills 365 days of ERA5 weather data for all 21 stations. The Lambda fetches one batched date-range request per station (21 total HTTP calls — not 365×21=7,665).

*[VI: Bước này nạp lại 365 ngày dữ liệu thời tiết ERA5 cho tất cả 21 trạm. Lambda thực hiện một yêu cầu phạm vi ngày cho mỗi trạm (21 lần gọi HTTP tổng cộng).]*

```bash
aws lambda invoke \
  --function-name openaq_weather_ingest \
  --payload '{"backfill_days": 365}' \
  --cli-binary-format raw-in-base64-out \
  /tmp/weather_response.json

cat /tmp/weather_response.json
```

Expected response:
```json
{"total_written": 7665, "errors": 0}
```

> **Timeout note:** With `backfill_days=365` the Lambda runs for approximately 90–120 seconds (within the 900-second timeout). Use `backfill_days=30` for a quick smoke test.

### Verify in S3

```bash
aws s3 ls s3://openaq-pipeline-yourname/raw/weather/ --recursive | wc -l
# Expect ~7,665 (21 stations × 365 days)
```

### Verify in Athena

```sql
SELECT location_id, COUNT(*) AS hourly_rows,
       MIN(date) AS earliest, MAX(date) AS latest
FROM openaq_raw.weather
GROUP BY location_id
ORDER BY location_id
LIMIT 5;
-- Each station should show ~8,760 rows (365 days × 24 hourly rows per day)
```

## 5.4.4 Validate in Athena | Kiểm tra trong Athena

Run these queries to confirm all three ingestion paths are working correctly before proceeding to dbt.

*[VI: Chạy các truy vấn này để xác nhận tất cả ba đường nạp dữ liệu đang hoạt động chính xác trước khi tiến hành dbt.]*

### Row count sanity checks

```sql
-- Batch: all 21 stations, 3 years
SELECT COUNT(*) AS total_rows FROM openaq_raw.batch;
-- Expected: ~900,000

-- Stream: last 60 days of near-real-time records
SELECT COUNT(*) AS stream_rows,
       MIN(datetime) AS earliest,
       MAX(datetime) AS latest
FROM openaq_raw.stream;
-- Expected: varies; latest should be within 30 minutes of now

-- Weather: full year for all stations
SELECT COUNT(*) AS weather_rows FROM openaq_raw.weather;
-- Expected: ~7,665 × 24 hourly records = ~183,960 rows
```

### Data quality checks

```sql
-- Sentinel value presence (should be 0 after staging, but check raw)
SELECT COUNT(*) AS sentinel_rows
FROM openaq_raw.batch
WHERE CAST(value AS DOUBLE) = -999.0;
-- Expected: ~34,000 rows in raw (3.8%); these are filtered at stg_measurements

-- 21 distinct station IDs
SELECT COUNT(DISTINCT locationid) AS station_count FROM openaq_raw.batch;
-- Expected: 21

-- Parameters present
SELECT DISTINCT parameter FROM openaq_raw.batch ORDER BY parameter;
-- Expected: bc, co, no2, o3, pm10, pm25, relativehumidity, so2, temperature, um003, um025
```

### Partition pruning proof

This is the key optimisation of the pipeline. Run both queries and compare **Data scanned**:

```sql
-- Full table scan (no date filter) — should be 0 bytes (Parquet footer only)
SELECT COUNT(*) FROM openaq_raw.batch;
-- Data scanned: 0 KB

-- Date-filtered query — should be <5 MB (only that month's partition)
SELECT COUNT(*), AVG(CAST(value AS DOUBLE)) AS avg_pm25
FROM openaq_raw.batch
WHERE year = 2025 AND month = 3 AND parameter = 'pm25';
-- Data scanned: < 5 MB  (not 800 MB)
```
