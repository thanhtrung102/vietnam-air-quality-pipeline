## 5.3.0 Build Lambda packages & choose your state backend | Đóng gói Lambda & chọn state backend

Two things must happen **before** `terraform init`/`apply`, or the apply will fail.

*[VI: Hai việc phải làm TRƯỚC `terraform init`/`apply`, nếu không apply sẽ lỗi: đóng gói Lambda zip và chọn state backend.]*

### 1. Build the five Lambda deployment zips

The Lambda functions deploy from pre-built zips referenced by `var.lambda_*_zip_path` (defaults
`../lambda/*.zip`). Terraform reads each zip's SHA-256 to detect changes — so the zips must exist
before `terraform plan`. Build them from the repo root:

```bash
bash lambda/build.sh
```

This writes `lambda/batch_sync.zip`, `streaming.zip`, `aqi_api.zip`, `completeness_check.zip`, and
`weather_ingest.zip` (all gitignored). Re-run it after any change to `lambda/` code.

> **If you skip this**, `terraform plan` fails with *"Invalid function argument … no file exists at
> ../lambda/batch_sync.zip"* from the `filebase64sha256()` call.

### 2. Choose a Terraform state backend

This repo's `terraform/main.tf` is pre-wired to the **maintainer's remote S3 state bucket**
(`openaq-tfstate-thanhtrung102`), which **you cannot access**. On a fresh clone, pick one:

- **(A) Local state — simplest, recommended for a solo reproduction.** Comment out the
  `backend "s3" { … }` block in `terraform/main.tf` (the ~6 lines inside `terraform { … }`).
  Terraform then keeps state in a local `terraform.tfstate` file. Nothing else to set up.
- **(B) Your own remote S3 state — for shared/team use.** Create a globally-unique, **versioned**,
  encrypted S3 bucket out-of-band, then replace the `bucket = "..."` value in the `backend "s3"`
  block with your bucket name. (TF ≥ 1.10 + `use_lockfile` needs no DynamoDB.) Example bootstrap:
  ```bash
  aws s3api create-bucket --bucket my-openaq-tfstate-$RANDOM \
    --region ap-southeast-1 --create-bucket-configuration LocationConstraint=ap-southeast-1
  aws s3api put-bucket-versioning --bucket <that-name> \
    --versioning-configuration Status=Enabled
  ```

> The rest of this workshop assumes **option A (local state)** unless noted. Either way, the AWS
> resources created are identical — only *where Terraform stores its state* differs.

## 5.3.1 Deploy with Terraform | Triển khai với Terraform

*[VI: Khởi tạo Terraform, xem xét kế hoạch triển khai và áp dụng để tạo khoảng 88 tài nguyên AWS cơ sở. Quá trình apply mất khoảng 4–6 phút.]*

### Initialize

```bash
cd terraform/
terraform init
```

Expected output (local state — option A):
```
Initializing the backend...
Initializing provider plugins...
- Finding hashicorp/aws versions matching "~> 5.0"...
- Installing hashicorp/aws v5.x.x...
Terraform has been successfully initialized!
```

> If you kept the S3 backend (option B), `terraform init` instead reports
> *"Successfully configured the backend 's3'"* and will prompt to copy any existing state. With a
> fresh bucket there is nothing to copy.

### Plan

```bash
terraform plan
```

Review the plan output. You should see roughly **88 resources to add, 0 to change, 0 to destroy** (the figure is lower than the QuickSight-enabled build because the QuickSight stack is disabled — see the note below). Key resources include:

| Resource | Description |

| `aws_s3_bucket.main` | Primary data lake bucket (`openaq-pipeline-{suffix}`) |

| `aws_s3_bucket_policy.main` | Public read scoped to `dashboard/` prefix only |

| `aws_glue_catalog_database.openaq_mart` | `openaq_mart` database |
| `aws_glue_catalog_table.openaq_batch` | `openaq_raw.batch` — OpenCSVSerde with partition projection |
| `aws_glue_catalog_table.openaq_stream` | `openaq_raw.stream` — NDJSON with partition projection |
| `aws_glue_catalog_table.weather` | `openaq_raw.weather` — NDJSON with partition projection |
| `aws_athena_workgroup.openaq` | `openaq_workgroup` — 10 GB/query scan limit |
| `aws_iam_role.lambda_exec` | `openaq_lambda_role` — shared Lambda execution role |
| `aws_kinesis_stream.openaq` | `openaq_stream` — ON_DEMAND capacity, 7-day retention |
| `aws_kinesis_firehose_delivery_stream.openaq` | `openaq_firehose` — GZIP delivery to `raw/stream/` |
| `aws_scheduler_schedule.*` | EventBridge rules for batch sync and streaming |

## 5.3.2 Verify Partition Projection in Athena | Kiểm tra Partition Projection

*[VI: Mở Athena console và chạy các truy vấn kiểm tra để xác nhận partition projection hoạt động đúng — không cần Crawler và dữ liệu quét bằng 0 KB trên bảng trống.]*

Open the **Athena console** → Workgroup: **`openaq_workgroup`** → Query editor.

<!-- TODO screenshot: Athena console → Query editor with openaq_workgroup selected, showing the database list in the left pane -->

### Test 1: Confirm table is queryable with zero data scanned

```sql
SELECT COUNT(*) FROM openaq_raw.batch;
```

Expected result:
- **Value:** `0` (no data synced yet — this is expected)
- **Data scanned:** `0 KB`

> If Athena returns an error like `HIVE_METASTORE_ERROR`, verify the Glue table was created and partition projection properties are set. Run `SHOW CREATE TABLE openaq_raw.batch;` to inspect.

### Test 2: Confirm Glue table schema

```sql
SHOW CREATE TABLE openaq_raw.batch;
```

The output should show `OpenCSVSerde` with partition projection annotations:

```sql
CREATE EXTERNAL TABLE `openaq_raw.batch`(
  `location_id` string,
  `sensors_id` string,
  ...
)
PARTITIONED BY (
  `locationid` int,
  `year` int,
  `month` int)
...
TBLPROPERTIES (
  'projection.enabled'='true',
  'projection.locationid.type'='enum',
  'projection.locationid.values'='2178988,2178989,...'
  ...
)
```

### Test 3: Confirm workgroup scan limit

Try a query that would normally scan the full table on an unpartitioned table:

```sql
SELECT parameter, COUNT(*) as cnt
FROM openaq_raw.batch
GROUP BY parameter;
```

With no data yet this returns 0 rows instantly. After [Step 5.4.1 (historical sync)](5.4-ingestion.md#batch), re-run this query with a date filter:

```sql
SELECT parameter, COUNT(*) as cnt
FROM openaq_raw.batch
WHERE year = 2025 AND month = 1
GROUP BY parameter;
```

Expected data scanned: **< 100 KB** (only January 2025 partition files read — not the full 800 MB archive).

## 5.3.3 Inject the OpenAQ API key into Secrets Manager | Nạp khóa API OpenAQ

`terraform apply` creates the `openaq/api_key` secret with a **`REPLACE_ME` placeholder** — it does
**not** contain your real key (there is no `openaq_api_key` Terraform variable; the key never lives in
state or tfvars). Inject the real key once, after every fresh apply, with the helper script:

*[VI: `terraform apply` tạo secret `openaq/api_key` với giá trị giữ chỗ `REPLACE_ME`. Chạy
`postdeploy.sh` một lần sau apply để nạp khóa thật vào Secrets Manager.]*

```bash
# from the terraform/ directory, after terraform apply completes
bash postdeploy.sh
# prompts (hidden input) for your OpenAQ v3 API key, then writes it to the secret
```

Get a free OpenAQ v3 API key at <https://openaq.org/developers>. The streaming Lambda reads the
secret at cold start, so the key is active on its next invocation ([Step 5.4.2](5.4-ingestion.md)).

> **If you skip this**, the secret stays `REPLACE_ME` and `openaq_streaming_producer` fails at runtime
> (the OpenAQ API rejects the placeholder). `postdeploy.sh` is idempotent — if the secret is already
> a real value it skips injection.
