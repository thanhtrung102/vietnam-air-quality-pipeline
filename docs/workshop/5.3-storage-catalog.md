# 5.3 Storage & Catalog Stack | Hạ tầng Lưu trữ và Danh mục

This section deploys all base AWS resources: the S3 data lake, Glue Data Catalog with partition projection, and Athena workgroup.

*[VI: Phần này triển khai tất cả tài nguyên AWS cơ sở: data lake S3, Glue Data Catalog với partition projection, và Athena workgroup.]*

## 5.3.1 Deploy with Terraform | Triển khai với Terraform

*[VI: Khởi tạo Terraform, xem xét kế hoạch triển khai và áp dụng để tạo toàn bộ ~40 tài nguyên AWS cơ sở. Quá trình apply mất khoảng 4–6 phút.]*

### Initialize

```bash
cd terraform/
terraform init
```

Expected output:
```
Initializing the backend...
Initializing provider plugins...
- Finding hashicorp/aws versions matching "~> 5.0"...
- Installing hashicorp/aws v5.x.x...
Terraform has been successfully initialized!
```

### Plan

```bash
terraform plan
```

Review the plan output. You should see **~93 resources to add, 0 to change, 0 to destroy**. Key resources include:

| Resource | Description |
|----------|-------------|
| `aws_s3_bucket.main` | Primary data lake bucket (`openaq-pipeline-{suffix}`) |
| `aws_s3_bucket_website_configuration.main` | Enables S3 static website endpoint for the Leaflet map |
| `aws_s3_bucket_policy.main` | Public read scoped to `dashboard/` prefix only |
| `aws_glue_catalog_database.openaq_raw` | `openaq_raw` database |
| `aws_glue_catalog_database.openaq_mart` | `openaq_mart` database |
| `aws_glue_catalog_table.openaq_batch` | `openaq_raw.batch` — OpenCSVSerde with partition projection |
| `aws_glue_catalog_table.openaq_stream` | `openaq_raw.stream` — NDJSON with partition projection |
| `aws_glue_catalog_table.weather` | `openaq_raw.weather` — NDJSON with partition projection |
| `aws_athena_workgroup.openaq` | `openaq_workgroup` — 10 GB/query scan limit |
| `aws_iam_role.lambda_exec` | `openaq_lambda_role` — shared Lambda execution role |
| `aws_kinesis_stream.openaq` | `openaq_stream` — ON_DEMAND capacity, 7-day retention |
| `aws_kinesis_firehose_delivery_stream.openaq` | `openaq_firehose` — GZIP delivery to `raw/stream/` |
| `aws_scheduler_schedule.*` | EventBridge rules for batch sync and streaming |
| `aws_sns_topic.openaq_alerts` | `openaq_alerts` — SNS topic for CloudWatch alarm emails |
| `aws_secretsmanager_secret.openaq_api_key` | Secrets Manager entry for `openaq/api_key` (7-day recovery window) |
| `aws_cloudwatch_metric_alarm.codebuild_failed` | `openaq-dbt-runner-failed` — alerts when dbt run fails |

```bash
terraform apply
```

## 5.3.2 Verify Partition Projection in Athena | Kiểm tra Partition Projection

*[VI: Mở Athena console và chạy các truy vấn kiểm tra để xác nhận partition projection hoạt động đúng — không cần Crawler và dữ liệu quét bằng 0 KB trên bảng trống.]*

Open the **Athena console** → Workgroup: **`openaq_workgroup`** → Query editor.

<!-- TODO screenshot: Athena console → Query editor with openaq_workgroup selected, showing the database list in the left pane -->

### Test 1: Confirm table is queryable with zero data scanned

```sql
SELECT COUNT(*) FROM openaq_raw.batch;
```

Expected result:
- **Value:** `0` (no data synced yet — this is expected)
- **Data scanned:** `0 KB`

> If Athena returns an error like `HIVE_METASTORE_ERROR`, verify the Glue table was created and partition projection properties are set. Run `SHOW CREATE TABLE openaq_raw.batch;` to inspect.

### Test 2: Confirm Glue table schema

```sql
SHOW CREATE TABLE openaq_raw.batch;
```

The output should show `OpenCSVSerde` with partition projection annotations:

```sql
CREATE EXTERNAL TABLE `openaq_raw.batch`(
  `location_id` string,
  `sensors_id` string,
  ...
)
PARTITIONED BY (
  `locationid` int,
  `year` int,
  `month` int)
...
TBLPROPERTIES (
  'projection.enabled'='true',
  'projection.locationid.type'='enum',
  'projection.locationid.values'='2178988,2178989,...'
  ...
)
```

### Test 3: Confirm workgroup scan limit

Try a query that would normally scan the full table on an unpartitioned table:

```sql
SELECT parameter, COUNT(*) as cnt
FROM openaq_raw.batch
GROUP BY parameter;
```

With no data yet this returns 0 rows instantly. After [Step 5.4.1 (historical sync)](5.4-ingestion.md#batch), re-run this query with a date filter:

```sql
SELECT parameter, COUNT(*) as cnt
FROM openaq_raw.batch
WHERE year = 2025 AND month = 1
GROUP BY parameter;
```

Expected data scanned: **< 100 KB** (only January 2025 partition files read — not the full 800 MB archive).
