# Speed up your Amazon Athena queries using partition projection

_Summarised in English from the AWS Big Data Blog post by Steven Wasserman, Janak Agarwal, Juan
Lamadrid, and Pathik Shah — 19 May 2021
([Amazon Athena](https://aws.amazon.com/blogs/big-data/category/analytics/amazon-athena/), Analytics).
Original article linked at the bottom; the write-up below is in my own words._

## The problem: metadata lookups on highly partitioned tables

Amazon Athena queries data in Amazon S3 and uses the AWS Glue Data Catalog (or a Hive metastore) to know
which partitions exist and where they live. For a table with only a handful of partitions this is
cheap. But as a table grows to thousands or tens of thousands of partitions — common for data
partitioned by year/month/day/hour — Athena has to fetch and filter all that partition metadata from
the catalog before it can read a single byte of data. That remote metadata step becomes the dominant
cost of the query, and it gets slower as more partitions accumulate.

## How partition projection works

Partition projection removes the catalog round-trip. Instead of looking partitions up, you describe
them once in the table's properties, and Athena **calculates** the partition values and their S3
locations in memory at query time. When a query has a `WHERE` clause on the partition columns, Athena
projects only the partitions that match — it never enumerates the whole catalog. Because in-memory
generation is far faster than a remote metadata call, queries against heavily partitioned tables speed
up, and newly arrived partitions are queryable immediately without `ALTER TABLE … ADD PARTITION` or a
crawler run.

## Projection types

You configure projection per partition column. The article describes these kinds:

- **`enum`** — a fixed list of values (e.g. a small set of regions or station IDs).
- **`integer`** — a numeric range, optionally with a fixed digit width.
- **`date`** — a date/time range with a format and an interval; ranges can be relative, such as a start
  of `2013-10-01` through `NOW`, which is ideal for time-series data that keeps growing.
- **`injected`** — values that aren't generated by Athena but are supplied directly in the query's
  `WHERE` clause, for high-cardinality keys that can't be enumerated.

A typical date-partitioned table sets table properties along the lines of
`projection.enabled = true`, a `projection.<col>.type`, range/format/interval settings, and a
`storage.location.template` that maps each projected value to its S3 prefix.

## Performance and cost

The post reports a customer example (Vertex) where projection cut a production query from 137 seconds
to about 10 seconds (~92%) and reduced month-end batch reporting from roughly 4.5 hours to 40 minutes
(~85%). Because projection lets Athena prune to exactly the needed prefixes — and pairs naturally with
columnar, compressed formats like Parquet — the volume of data scanned drops too, which directly lowers
Athena's per-query cost.

## When to use it, and the main limitation

Partition projection is the right tool when a table is highly partitioned, when new date/time
partitions appear on a schedule, or when keeping the catalog in sync would otherwise be a chore. The
key limitation: the projected partitions are understood **only by Athena**. Other engines that read the
same table (Redshift Spectrum, EMR) still rely on conventional partition metadata in the catalog.

## Applied in this project

The Vietnam Air Quality pipeline catalogs **three raw external tables** — `batch`, `stream`, and
`weather` — and every one of them uses **partition projection over date/time keys**, so **no Glue
crawler ever runs** and there is no per-scan crawl cost. New daily and hourly objects land in S3 under
Hive-style prefixes and become queryable immediately, with nothing to register. Paired with the
`openaq_workgroup` **10 GB per-query scan cap**, projection is a big part of why the whole pipeline
stays inside its ~$3–8/month envelope. The streaming objects projection reads are produced by the
Firehose path described in [Blog 2](../3.2-firehose-dynamic-partitioning/).
