**Objective —** stand up the storage and query foundation: the S3 data lake, the Glue catalog
(partition projection, no crawler), and the cost-capped Athena workgroup.

This layer provisions the S3 data lake, the Glue catalog (partition projection — no crawler), and the
Athena workgroup. It is created by the same `terraform apply` as the rest of the stack.

## Deploy
```bash
cd terraform
terraform init        # configures the remote S3 backend (versioned, SSE, native lockfile)
terraform plan        # review — a fresh clone shows ~82 resources to add (forecast gated off)
terraform apply
```

## What gets created
- **S3 bucket** `openaq-pipeline-<suffix>` — versioning on, AES256 + bucket-key SSE, static website
  (`dashboard/*` public), lifecycle: `athena-results/` expire 7 d, `raw/stream/` 60 d, noncurrent 7 d,
  `processed/` → Intelligent-Tiering.
- **Glue databases** `openaq_raw` + `openaq_mart`; 3 external raw tables (`batch`, `stream`, `weather`)
  with **partition projection** — no crawler runs, so there is no per-scan crawl cost.
- **Athena workgroup** `openaq_workgroup` — results to `s3://…/athena-results/`, SSE_S3, **10 GB
  per-query scan cap** (cost guardrail).

## Verify
```bash
aws glue get-databases --query 'DatabaseList[?starts_with(Name,`openaq`)].Name'
aws athena get-work-group --work-group openaq_workgroup --query 'WorkGroup.State'
```

{{% notice info %}}
**Expected output.** At this stage the first command lists the two catalog databases —
`[ "openaq_raw", "openaq_mart" ]` — and the second prints `"ENABLED"`, confirming the 10 GB per-query
scan cap is in force. (Once dbt runs in **5.5**, the same query also returns its auto-created
`openaq_mart_dbt_test__audit` and `openaq_mart_openaq_processed` schemas — four names in total.)
{{% /notice %}}
